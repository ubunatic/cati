# 002 — Browser Input: q/ESC/^C not working in browser + video viewer

**Status:** ✅ Closed  
**Refs:** [Browser.md](../docs/Browser.md#3-double-buffered-raw-mode)

---

## Symptoms

1. `q` and `ESC` had no effect in the browser grid.
2. `^C` during video playback stopped the video but left the browser running instead of exiting.
3. The single-file `--play` mode worked fine — isolating the issue to the browser path.

## Root Causes

### 1. `interactiveVideo` ran in cooked terminal mode

The browser's Enter handler restores the terminal to cooked mode before launching any viewer
(`term.Restore`), relying on each viewer to set its own raw mode. `interactiveWithChan` did this
correctly (it calls `term.MakeRaw` internally). `interactiveVideo` did **not** — it was never
updated to match.

In cooked mode `os.Stdin.Read` blocks until a full line is submitted (i.e. the user must press
Enter after `q`). Since the shared goroutine was in cooked-mode blocking, no single-key tokens
were produced, so `q` and `ESC` appeared completely broken.

`^C` still worked because SIGINT is generated by the terminal driver regardless of line-discipline
mode — it never flows through the stdin byte stream.

### 2. SIGINT not propagated from viewer back to browser

Both `browser()` and `interactiveVideo()` register their own `signal.Notify` channels for
SIGINT. Go delivers the signal to **all** registered channels, so the browser's `sigs` channel
received the SIGINT while `interactiveVideo` was running. However, the browser was blocked inside
its `inputs` case handler waiting for `interactiveVideo` to return. After the viewer returned,
the main `for { select {} }` loop resumed — but only after redraw and a full event drain, at
which point `<-sigs` competed with other cases and was not reliably selected immediately.

The result: pressing `^C` stopped the video (via the video's sigs channel) but the browser
continued running.

## Fixes Applied

**`cmd/interactive.go` — `interactiveVideo`:**
Added `term.MakeRaw` / `term.Restore` at the top of the function, matching `interactiveWithChan`.
This ensures the terminal is in raw mode during video playback so single-key presses are
immediately readable by the shared stdin goroutine.

**`cmd/browser.go` — Enter handler:**
After the viewer returns, explicitly drain the browser's `sigs` channel:
```go
select {
case <-sigs:
    shouldQuit = true
    return
default:
}
```
This guarantees that a `^C` pressed during viewing exits the browser immediately on return,
regardless of what the outer select would have chosen next.

## Lessons

- Any function that takes over the terminal and reads stdin needs to call `term.MakeRaw`
  itself, even when a shared input channel is used — the channel does not imply raw mode.
- When multiple goroutines register `signal.Notify` for the same signal, Go delivers to all,
  but the *consumer* of each channel must be active to drain it. Explicitly checking the channel
  after returning from a blocking sub-operation is more reliable than relying on `select` priority.
- The single-file `--play` path (which reads stdin bytes directly, one byte at a time, and
  never goes through the cooked-mode restore cycle) was a useful reference implementation that
  exposed the divergence.
